## What the deployed app is

A **read-only artifact server** plus a static-ish web client. The deployed API serves committed
artifacts only — no runtime data fetching, no live retrieval, no credentialed access to anything
(cardinal rule 1). The integrity contract is `artifacts.manifest.json` (SHA-256 of every served
committed artifact, keyed by serving path), regenerated by
`uv run python scripts/build_artifact_manifest.py` and staleness-guarded by
`apps/api/tests/test_artifact_manifest.py`. Stage D's live verifier checks the deployed instance
against this manifest at the SHA reported by `/api/version`.

## Environment schema

The single schema lives in `.env.example`. Semantics:

| Variable | Required | Meaning / failure mode |
|---|---|---|
| `AETHER_ENV` | no (default `development`) | `production` switches on the prod-required checks below. Invalid values fail startup (`ConfigError`). |
| `AETHER_ALLOWED_ORIGINS` | **in production** | Comma-separated exact CORS origins. No `*`, no wildcard patterns, no trailing slash — all rejected at startup. Missing in production = **loud startup failure**, never a silent localhost default. Production value: `https://aether.arkaneworks.co`. |
| `AETHER_GIT_SHA` | **in production** | Baked as a Docker **build arg** → env (`docker build --build-arg GIT_SHA=$(git rev-parse HEAD)`); never a runtime `git` call. `/api/version` reports it; the web footer renders it (fetched, not hardcoded). Dev reports an honest `dev`. |
| `AETHER_DATA_ROOT` | no | Override for the artifact tree root (the image bakes `/app/data`). A path, not a secret. |
| `NEXT_PUBLIC_API_BASE` | for prod web build | Inlined into the client bundle at `next build` — changing it requires a rebuild. **Public by design.** |
| `NEXT_PUBLIC_CESIUM_ION_TOKEN` | no (NOT provisioned, per gate) | Client-visible by design if ever used; URL-restrict in the ion dashboard. Default is token-free ESRI World Imagery. |

Platform-side secrets (never in the repo, image, or logs; created and entered by the **[Human]**
directly in platform dashboards): `FLY_API_TOKEN` (GitHub Actions secret, scoped deploy token) —
that is the complete list. Vercel's native Git integration needs no repo-side secret.

## Security headers (what and why)

Set on every API response by middleware (`aether_api/main.py`):

- `X-Content-Type-Options: nosniff` — the API serves PNG/GeoJSON/JSON artifacts; browsers must not
  MIME-sniff them into something executable.
- `X-Frame-Options: DENY` — the API has no embeddable UI; framing it serves only clickjacking.
- `Referrer-Policy: no-referrer` — a read-only artifact server needs no referrer context.

Not set, deliberately: `Strict-Transport-Security` (Fly terminates TLS and serving is
HTTPS-only at the edge; HSTS on the API hostname is platform-managed), CSP (no HTML UI beyond the
FastAPI docs pages, which ship their own assets).

## FastAPI docs endpoints — ON in production (decision)

`/docs`, `/redoc`, `/openapi.json` stay enabled. Reasons, as argued at the Stage A gate and
ratified (F7): the API is read-only and serves only public, license-audited artifacts; the OpenAPI
surface documents exactly the committed-artifact contract; transparency about what the API serves
is part of the portfolio value. There is no auth to leak and no mutating route to discover (the
read-only property is guard-tested, not conventional).

## Uvicorn production posture

The container runs `uvicorn aether_api.main:app --host 0.0.0.0 --port 8080 --proxy-headers
--no-server-header` — single worker (a 256 MB always-on machine serving committed files needs no
more; scale-out is a Fly machine count decision, not a worker count one), proxy headers honored
because Fly's edge terminates TLS, no `Server:` banner.

### Pre-deploy invariant (Stage B gate addition)

Before any deploy, the **image-inventory guard** must be green on the image being shipped:
`uv run python tools/verify_image_inventory.py <image>` — every file under `/app/data` committed
at the baked SHA (positive subset check; `.dockerignore` is belt, this is suspenders). For
`--remote-only` deploys the same invariant is enforced by deploying only from a clean, pushed
checkout (CI-verified SHA) — and the guard runs against a locally built twin of the same SHA.

## Stage D — deployed-integrity verifier

`tools/verify_deployment.py <base_url>` proves the live API serves byte-identical committed
artifacts **at the SHA it reports** — the pin is the guard's validity, so a stale deploy is
distinguishable from drift. It (a) reads `/api/version` for the deployed SHA; (b) reads the repo
**at that SHA** via `git show` (never the working tree); (c) hashes the **transport-decoded** body
of every raw endpoint against the manifest at the pinned SHA, over **two transport paths** —
default (server may gzip/br; decoded) and `Accept-Encoding: identity` — failing red with the first
differing byte offset on any mismatch; (d) deep-equals every composed endpoint against a
reconstruction built by running the **pinned-SHA code over the pinned-SHA data** (`git archive` +
PYTHONPATH-imported, asserted to load from the extract); (e) asserts the negative space — NOAA ISD
raw, the route-unreachable `_nasa_k` siblings, composed-source files, and non-whitelisted
events/layers all 404.

CI: `.github/workflows/verify-deployment.yml` (scheduled + manual; chain after an automated deploy
once it exists) runs the verifier against production and **asserts the Fly machine count == 1**
(the Stage C auto-HA re-add finding). It needs `fetch-depth: 0` — the deployed SHA is often an
ancestor of HEAD.

### Failure semantics

| Result | Meaning | Exit | CI |
|---|---|---|---|
| **GREEN** | Pinned SHA == main HEAD and every check passes — the deployed instance is provably the committed one. | 0 | pass |
| **WARNING** | Every integrity check passes, but the deploy is **not** main HEAD: either *stale* (pinned SHA is an ancestor of HEAD — e.g. a web-only fix advanced main without an API redeploy) or *diverged* (not an ancestor). Internally consistent; **not a drift.** | 0 | pass + `::warning::` |
| **RED** | A raw byte mismatch (after transport decoding — i.e. real platform transformation or deploy drift), a composed deep-equality failure, a negative-space leak, or an **unverifiable pin** (the deployed SHA is absent from local history, or no manifest at that SHA). | 1 | fail |

**The SHA pin is what separates WARNING from RED.** A stale-but-consistent deploy passes every
byte check *because it is checked against its own SHA's manifest* — so "the deploy is old" surfaces
as a warning, while "the deploy serves bytes that differ from what that SHA committed" is red. Use
`--strict` to make WARNING fail too (a lane that demands the deploy be exactly main HEAD).

A current WARNING is expected and benign: the Stage C UI fix advanced `main` (web redeployed by
Vercel) while the Fly API stayed at its prior SHA. It clears the moment the API is next deployed —
e.g. the first automated deploy after the human creates `FLY_API_TOKEN` (below).
